Releasing Helm Charts

Most of the Helm charts I’ve written aren’t for deploying apps of my own. They’re primarily abstractions of complicated kubernetes resources that give platform consumers an opinionated, sane path to deploy their workloads onto AKS without every team reinventing a Deployment, a Service, an ingress and the rest. So for me that changes what “releasing” a chart well actually means and also simplifies the development process for charts.

One chart per repo

I keep one chart in one repository. This is the same argument I make for conventional commits and automated releases: when a repo holds exactly one releasable thing, the commit history is that chart’s history, the changelog is honest and semantic-release can version it without any monorepo gymnastics. A consumer looking at the repo sees the chart and nothing else.

Inside that repo it’s a normal chart. The thing I like to add is a folder of values files that stand in for the ways consumers actually use it, this gives me both something to test against locally during development and the CI something concrete to validate. Helm has a convention for this, a ci/ directory of *-values.yaml files.

A schema that rejects what it doesn’t use

In my opinion, the single highest-value thing you can add to a chart is a values.schema.json. Helm validates supplied values against it on template and install, so whoever passes the wrong type or misses a required field gets a clear error at the boundary instead of a confusing template crash three layers in.

The part I consider essential is additionalProperties: false wherever you can manage it. A values key that the chart doesn’t read is one of the most baffling things a consumer can hit, because it fails silently. They set replicaCount, the chart expected replicas, nothing errors and nothing changes. Rejecting unknown keys turns that quiet no-op into an instant and obvious failure.

{
  "$schema": "https://json-schema.org/draft-07/schema",
  "type": "object",
  "additionalProperties": false,
  "required": ["replicas", "image"],
  "properties": {
    "replicas": { "type": "integer", "minimum": 1 },
    "image": {
      "type": "object",
      "additionalProperties": false,
      "required": ["repository", "tag"],
      "properties": {
        "repository": { "type": "string" },
        "tag": { "type": "string" }
      }
    }
  }
}

The schema is the consumer-facing contract. If a change to it would break someone’s existing values, that’s a major, which is exactly the kind of thing the release machinery further down has to get right.

Test what it renders, not what it installs

I keep chart testing deliberately lightweight. I don’t spin up a cluster, I check that the chart renders valid Kubernetes for every values file I care about. That catches the overwhelming majority of mistakes for almost no cost or flakiness.

Two steps do the work. helm lint for the static checks, then helm template against each values file in ci/, piped into kubeconform to validate the rendered manifests against the Kubernetes schemas.

helm lint .

for f in ci/*-values.yaml; do
  helm template . --values "$f" | kubeconform -strict -summary
done

This is where those values files earn their keep. They’re not just a local convenience, they’re the test matrix. Every combination a consumer might realistically pass becomes a case that has to render cleanly before anything ships.

There’s another important test worth adding and it comes from the templating nature of Helm itself. What I mean is a small change to a conditional or a loop can add, remove or reshape whole resources once it’s rendered and you’d never judge the size of it from the diff of the chart source code alone. To catch that, render the previously released chart and the version in your PR with the same values, then diff the output. What you’re looking at is the real change, the actual Kubernetes a consumer ends up with, rather than the templating that produced it.

I treat that diff as information, not a gate. It goes in front of whoever’s reviewing the change so they can see the blast radius, instead of automatically failing the build on any difference because there’s always going to be a difference and that’s the whole point of a release. There is the case where old values files won’t render against the new chart and therefore there’s no clean diff to produce but I don’t fight that. The old inputs failing to render is itself a breaking change made visible and that’s exactly when the changelog and a human reviewer matter most.

You can go further still and install into an ephemeral cluster with helm test hooks but I’ve found static rendering, the schema and a render diff catch enough that I’ve never needed to.

Versioning and publishing

The version is derived, not typed, the same as it is for everything else I release. semantic-release reads the commits, decides the bump and the rest of the release falls out of that. The one Helm-specific wrinkle is that the chart version lives in Chart.yaml, and I don’t want anyone hand-editing it, so I let semantic-release write it at package time.

semantic-release defaults to publishing npm packages, so for a chart you drive the actual commands through @semantic-release/exec. It packages the chart with the computed version and pushes it to the registry.

{
  "plugins": [
    "@semantic-release/commit-analyzer",
    "@semantic-release/release-notes-generator",
    "@semantic-release/changelog",
    ["@semantic-release/exec", {
      "publishCmd": "helm package . --version ${nextRelease.version} && helm push *.tgz oci://acme.azurecr.io/helm"
    }]
  ]
}

I deliberately don’t set an appVersion on these charts. appVersion is for tracking the application image a chart ships and my charts don’t ship one. They’re deployment abstractions so the chart’s own version is the only version that means anything and tying it to a phantom app version would just be noise.

For the registry I’ve used both Artifactory and Azure Container Registry. Both speak OCI, which since Helm 3.8 is how charts are meant to be published: helm push straight into the registry as an OCI artifact.

The point

A chart that other teams build on has to behave like the product it is. Pin it to a derived version, lock the values down with a schema so misuse fails loudly, validate what it renders and publish it to an OCI registry with a changelog that tells consumers what changed. None of it is heavy. It’s the same release discipline I follow everywhere and the payoff is a consumer who can take a new version on purpose, knowing what they’re getting, instead of hoping it works.
