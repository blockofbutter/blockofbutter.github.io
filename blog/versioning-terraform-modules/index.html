Versioning Terraform Modules

To set the scene, I was making a small change to some Terraform configuration; new tags on a resource. I ran a terraform plan via CI and found a list of changes I didn’t expect including a resource that needed to be destroyed and recreated. None of it related to what I actually wanted to change. How could this happen?

My reflex was to look at my own diff. So I did. I reviewed the PR and the changed files and ran a git log on the configuration and it was all clean. Whatever was causing the unexpected changes didn’t exist anywhere in my repository. That’s because it hadn’t come from my repository at all. It came from a module pinned like this:

module "network" {
  source = "git::https://git.example.com/modules/network.git?ref=main"
}

Terraform resolves module sources at terraform init, not at plan or apply so pinning to main means “whatever is in that branch when I run init.” Someone had merged a change to the module days earlier, my init quietly pulled it and the next plan reflected it. The only way I found it was by digging through the module’s own source and change history. Nothing in my own history would ever have told me.

So my files didn’t change and my blame was clean, yet the plan was different. The thing that changed lived outside my repo and got pulled in silently. That’s why version pinning matters and it’s how you stop this.

What a version actually promises

It’s tempting to think of a module version the way you think of a library version: the interface is the variables and outputs and a breaking change is a renamed variable. That’s part of it but in Terraform the contract is wider because a module also owns a piece of state.

A change that doesn’t touch a single variable can still be breaking. Rename a resource, change the key in a for_each, or bump a required provider and the next plan wants to destroy and recreate real infrastructure. The interface is unchanged and the blast radius is enormous.

So when I talk about a breaking change in a module, I mean anything that forces replacement or changes the state shape, not just anything that changes the variable signature. That definition matters because it’s what an honest version bump has to capture.

How you pin, and what each pin is worth

I’ve published modules through plain git, through Terraform Enterprise’s private registry and through Artifactory’s Terraform registry. They’re not equivalent and the differences are the whole point.

Git refs: tags and commit SHAs

With a git source you pin through the ref query parameter. You can point it at a tag, a branch or a commit SHA.

# Readable, but a tag is mutable: it can be force-pushed to point elsewhere.
source = "git::https://git.example.com/modules/network.git?ref=v1.4.0"

# Immutable, but it tells you nothing without going and looking it up.
source = "git::https://git.example.com/modules/network.git?ref=9f3c1ab"

A SHA is the only genuinely immutable thing here. It’ll always resolve to the same code. The trouble is that it’s not human readable. When I open a configuration I want to know that I’m on v1.4.0 and that the latest is v1.6.0, so I know roughly how far behind I am and whether the gap is a patch or two majors. A SHA gives me none of that at a glance and a configuration full of them is a configuration nobody upgrades because nobody can tell what they’re looking at.

So personally I like to pin to a version, not a SHA. The catch is that a git tag is mutable. ?ref=v1.4.0 is a convention that everyone agrees not to move that tag, not a guarantee that it can’t move. That’s the gap a registry closes.

As a side note, it’s quite nice using Git refs as you don’t have to worry about publishing the module.

A module registry

Publish the same module through a registry and you get the version argument and can make the most of Terraform’s constraint operators. The version also becomes immutable by construction because generally the registry refuses to let you republish one that already exists.

module "network" {
  source  = "app.terraform.io/acme/network/azurerm"
  version = "~> 1.4"
}

Artifactory speaks the same registry protocol so the shape is identical and only the host changes. This is where the readable-version preference and the registry stop being two opinions and become one. I want to read my pins and I want those pins to be immutable. A registry gives me both: version = "1.4.0" is human readable and the registry won’t let 1.4.0 ever mean anything else.

There is no module lockfile

Worth saying plainly, because it surprises people: .terraform.lock.hcl locks providers only. It doesn’t lock modules. There’s nothing that records and verifies the module versions you resolved last time. The source and version lines in your configuration are the entire contract. That’s why the pin isn’t a nicety. It’s the only thing standing between you and the problem I encountered at the top of this post.

Why not pin and forget

If a SHA or an exact version is so safe then the seemingly obvious move is to pin everything to an exact version forever and never think about it again. I’ve worked in teams that did exactly that mainly because of higher priorities but that comes with a bag of troubles down the line.

One of my biggest fears is nothing gets upgraded. The pins are exact and the configurations are stable but two years later you’re several majors behind on everything, the security patches never landed and the upgrade you can no longer avoid is huge because you’re crossing all those breaking changes at once. Pinning is easy. Upgrading the pins is the actual work and over-pinning just hides that work until it’s too big to do safely.

Deciding the version

If consumers are going to pin to versions, those versions have to mean something and deciding the number by hand is where discipline quietly erodes. Enter my favourite tool for releases that I use elsewhere: conventional commits drive the bump and a release runs on merge to tag the version, generate the changelog and publish to the registry. A fix: is a patch, a feat: is a minor and a feat!: or a BREAKING CHANGE: footer is a major.

feat!: switch network module to for_each over count

BREAKING CHANGE: subnets are now keyed by name instead of index,
so existing subnets will be replaced. Review the plan before applying.

There’s one Terraform-shaped trap in this. Conventional commits describe the author’s intent about the code but a breaking change in a module is about state impact. A commit can be a tidy internal refactor by every code-review standard and still force every consumer to replace resources. If the author labels that a fix:, the automation cuts a patch release, consumers take it automatically and the tooling has confidently lied to them. So the rule module authors need to keep in mind is simple: if your change forces replacement or changes the state shape, it’s a major, no matter how small the diff looks. The version number is a promise about what happens on apply, not about how much code moved.

Making upgrades easy

Pinning protects you. The rest of the work is making sure you can move off a pin without it being a project. A few things that earn their place:

  • Constraints over exact pins, on the registry. version = "~> 1.4" takes patches and minors automatically while holding the major. You get fixes without a manual bump and the immutable-version guarantee means “automatic” isn’t the same as “unpredictable.” Git refs can’t express this which is another reason hosting modules in a registry wins.
  • Automate the upgrade, not just the pin. Renovate and Dependabot both understand Terraform module sources. They open a pull request for each bump, CI runs plan and you review the diff before anything merges. That turns the upgrade treadmill into a stream of small, reviewable changes instead of one annual cliff.
  • Tag only tested versions. A published version should be one that passed its tests, so that a green pin is a known-good pin rather than just the latest commit that happened to build.
  • Read the changelog before you apply. The whole reason to generate one from the commits is so a consumer can see what a bump means before taking it. A major with a clear “subnets will be replaced” note is a bump you take on purpose in a maintenance window, not one you discover in a plan.

The point

Pinning was never the goal. Easy, confident upgrades are. Pin to a readable version, publish those versions somewhere that makes them immutable and put enough release discipline behind the number that consumers can trust it. Then upgrading is a quick win you choose to do and not dread every few months.
