Conventional Commits to Automated Releases

I really don’t enjoy bumping versions by hand. Take a Helm chart: the version lives in Chart.yaml but then you also tag the repository with that same version so the release is traceable. That’s the same number typed by hand in two places every release. Terraform modules are no different: the tag is the version and it’s on you to pick the right one.

It sounds trivial until you remember we’re human. You forget to bump the tag after editing the chart. You write v1.2 instead of v1.2.0, or 1.20 when you meant 1.2.0. You ship a breaking change as a patch because you were in a hurry. None of these are dramatic on their own, but they’re exactly the kind of small repetitive step a person eventually gets wrong and a machine never does. So I’d rather not be the one deciding the number at all.

That’s what conventional commits and semantic-release give you.

What conventional commits actually are

A conventional commit is just a commit message with a small bit of structure at the front:

fix: correct the default memory limit on the worker deployment
feat: add support for pod topology spread constraints
feat!: drop the legacy ingress annotations

The prefix is the contract. A fix: is a patch, a feat: is a minor and a feat!: (or a BREAKING CHANGE: footer) is a major. That maps directly onto semantic versioning, which is the whole point: the commit already tells you how the version should move, so nobody has to decide it separately.

This is the bit that changes how you think about a commit message. It’s no longer just a note to a human reading the log later. It’s an input to a machine that’s going to cut a release from it.

From a commit to a release

semantic-release is the tool I reach for. It runs in CI on a merge to your release branch and does the whole sequence for you: it reads every commit since the last release, works out the highest bump implied by them, decides the next version, generates the changelog, tags the repo and publishes the artifact.

So a couple of fixes and one feat: will change your version from v1.3.0 to v1.4.0, all worked out and tagged without anyone typing it. Not every commit forces a release either. A docs:, chore: or ci: change bumps nothing, which is exactly what you want, since fixing a typo in the README shouldn’t ship a new version of the chart.

The number is never typed. It’s computed from the commits so the git tag, the changelog and the version baked into the artifact all come from the same source of truth. The double-entry problem from the intro just disappears because there’s only one entry now and it’s the commit message you were going to write anyway. This also makes the commit history extremely clean and simple to read.

The changelog is worth dwelling on, because it’s the part your consumers actually read. @semantic-release/release-notes-generator groups the commits into release notes under features, fixes and breaking changes, and @semantic-release/changelog writes them into a CHANGELOG.md that lives in the repo. The notes come straight from the commit messages, so anyone consuming the artifact can look at a version and see what’s in it before they upgrade rather than guessing from a bare tag. In my opinion that’s a real gift to whoever is on the other end of your release.

The flip side is that your commit messages are load-bearing. If they’re sloppy, the release is wrong and you find out by shipping a bad version rather than by getting told off in review. Which leads to the two opinions I hold most strongly about running this in anger.

Polyrepo! not monorepo

semantic-release assumes one releasable artifact per repository. It versions the repo as a unit, so the natural fit is one artifact per repo. There are monorepo plugins but they add real complexity from my investigation and the clean path is a polyrepo layout.

You can read that as a downside, a push away from the monorepo most teams default to but I read it as a feature. When a repo holds exactly one artifact, the commit history is that artifact’s history. Every commit affected this thing and nothing else so the changelog is honest and the blast radius of any change is obvious. You never have to sit in a giant monorepo log trying to work out which commit touched which of forty artifacts because that question can’t even be asked here. The version and the history line up with the thing you actually ship.

Treat the commit message as an input, and validate it

If the commit message is an input to your release machinery, validate it like any other input. There are two places to do that and I like both.

The first is CI. A pull request whose commits don’t parse as conventional commits should fail, the same as a failing test or a lint error would. commitlint checks messages against the conventional ruleset:

// commitlint.config.js
export default {
  extends: ["@commitlint/config-conventional"],
};

The second is a hook on your own machine so you catch a bad message before it’s ever pushed. This is shifting left in the most literal sense: the feedback lands at the moment you write the commit, not minutes later in a failed pipeline. A commit-msg hook wired up with husky does it:

# .husky/commit-msg
npx --no-install commitlint --edit "$1"

CI is the gate that protects the release. The local hook is the courtesy that stops you wasting a CI run on a typo. You should run the gate regardless because hooks can be skipped (I concede I’ve done it) but the hook is what makes the workflow pleasant day to day.

One caveat that catches people out: if you squash-merge pull requests, none of this per-commit checking matters at release time because semantic-release only ever sees the single squash commit. Its message, which most platforms default to the PR title, is the one that drives the version. So if you squash, lint the PR title as well or just don’t allow squashes.

Derive, don’t type

Manual versioning isn’t hard, it’s just the kind of repetitive, format-sensitive task people quietly get wrong. Conventional commits turn the message you already write into the single source of truth for the version. semantic-release turns that into a tagged, changelogged release without anyone picking a number. And validating the messages, in CI and ideally at commit time too, keeps the one input the whole thing depends on clean.

You can see this tool put into practice when publishing versioned Helm charts and for tagging Terraform modules so consumers can pin them sanely. If you want the underlying rules, the Conventional Commits spec and Semantic Versioning are both short and worth checking out.
